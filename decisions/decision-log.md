# 똑독(Knockdog) 의사결정 기록

> 이 문서는 사용자가 직접 확정했거나 승인한 제품 의사결정만 기록한다.  
> 제안, 아이디어, 가설은 확정 결정과 구분한다.

---

## 상태 정의

- **확정:** 현재 제품 판단의 기준으로 사용
- **검토 중:** 논의 중이며 아직 기준으로 사용하지 않음
- **가설:** 검증이 필요한 추정
- **폐기:** 현재 사용하지 않는 결정
- **대체됨:** 더 새로운 결정으로 교체됨

---

# 확정 결정

## 2026-06-17 — 현재 제품 단계는 MVP로 본다

- 상태: 확정
- 결정: 똑독의 현재 제품 단계는 MVP로 정의한다.
- 이유: 기능 완성도 확대보다 실제 사용자 확보와 핵심 가치 검증이 우선이기 때문이다.
- 근거: 사용자가 직접 현재 단계를 MVP라고 명시했다.
- 영향:
  - 기능 수보다 가입과 첫 사용 흐름을 우선한다.
  - 미래 확장 기능은 기본적으로 후순위다.
  - 작은 실험과 직접 온보딩을 선호한다.
- 재검토 조건:
  - 반복 사용자가 확보되고 다음 성장 단계의 목표가 확정될 때

---

## 2026-06-17 — 최상위 목표는 첫 실제 사용자 확보다

- 상태: 확정
- 결정: 현재 가장 중요한 목표는 단 한 명이라도 실제 사용자를 확보하는 것이다.
- 이유: 제품 가설을 실제 사용으로 검증하는 것이 현재 단계의 최우선 과제이기 때문이다.
- 근거: 사용자가 직접 목표를 명시했다.
- 영향:
  - 사용자 확보에 직접 연결되지 않는 기능은 우선순위를 낮춘다.
  - 제품 개발 외에 직접 영업, 초대, 수동 온보딩도 해결책으로 검토한다.
  - 첫 사용자의 실제 행동을 다음 의사결정의 핵심 근거로 사용한다.
- 재검토 조건:
  - 첫 사용자 확보 후 다음 목표가 정해질 때

---

## 2026-06-17 — 현재 핵심 KPI는 가입 수다

- 상태: 확정
- 결정: 현재 제품 의사결정의 핵심 KPI는 가입 수다.
- 이유: 첫 사용자 확보 여부를 가장 직접적으로 확인하기 위해서다.
- 근거: 사용자가 직접 가입 수를 핵심 지표로 명시했다.
- 영향:
  - 가입 시작, 완료, 초대 수락 등 세부 정의를 추가로 정해야 한다.
  - 가입에 직접 영향을 주지 않는 작업은 후순위로 판단할 수 있다.
- 미확인:
  - 가입 완료 이벤트의 정확한 정의
  - 원장 가입과 견주 가입 중 우선 측정 대상
  - 테스트 계정 제외 기준
- 재검토 조건:
  - 가입자가 확보된 후 활성화 또는 유지 지표가 더 중요해질 때

---

## 2026-06-17 — 핵심 제품 위험은 원장 또는 견주의 접근 어려움이다

- 상태: 확정
- 결정: 유치원 원장 또는 견주가 서비스에 접근하고 가입하는 데 어려움이 있을 수 있다는 점을 핵심 위험으로 본다.
- 이유: 접근 단계에서 막히면 첫 사용자 확보와 가입 수 목표를 달성할 수 없기 때문이다.
- 근거: 사용자가 직접 우려 사항으로 명시했다.
- 영향:
  - 회원가입, 초대, 인증, 연결, 첫 사용 흐름을 우선 검토한다.
  - 사용자 특성은 추측하지 않고 실제 가입 과정을 관찰한다.
- 미확인:
  - 실제로 가장 크게 막히는 단계
  - 원장과 견주 중 어느 쪽의 문제가 더 큰지
- 재검토 조건:
  - 실제 사용자 관찰과 가입 데이터가 확보될 때

---

## 2026-06-17 — AI PM의 가장 중요한 책임은 우선순위 결정이다

- 상태: 확정
- 결정: AI Product Manager의 핵심 책임을 기능 제안이 아니라 우선순위 결정으로 둔다.
- 이유: 제한된 리소스를 현재 가장 중요한 문제에 집중하기 위해서다.
- 근거: 사용자가 가장 중요한 능력으로 우선순위 결정을 명시했다.
- 영향:
  - AI는 여러 선택지를 나열하는 데 그치지 않는다.
  - 가장 먼저 할 한 가지를 지정한다.
  - 지금 하지 않을 일도 명확히 한다.
  - 사용자의 제안에도 필요하면 반대한다.
- 재검토 조건:
  - 제품 단계와 AI 역할이 변경될 때

---

## 2026-06-17 — AI 저장소의 V1 범위는 기획과 디자인으로 제한한다

- 상태: 확정
- 결정: `knockdog-ai` V1은 제품 기획과 UX/UI 디자인 역할만 다룬다.
- 이유: 현재 목적이 코드 구현이 아니라 제품 우선순위와 사용자 경험 판단이기 때문이다.
- 근거: 사용자와의 대화에서 기획·디자인만 포함한 단순 구조를 선택했다.
- 영향:
  - 백엔드, 프론트엔드, QA, 마케팅 역할은 V1 기본 범위에서 제외한다.
  - 기술 구현은 별도 코드 저장소에서 수행한다.
- 재검토 조건:
  - V1 문서 운영이 안정되고 다른 역할이 실제로 필요해질 때

---

## 2026-06-17 — AI 문서는 별도 저장소에서 관리한다

- 상태: 확정
- 결정: 기획·디자인 AI 문서를 `knockdog-ai` 같은 별도 Git 저장소에 축적한다.
- 이유: 코드 저장소와 기획 지식 저장소의 목적이 다르며, 독립적으로 관리하는 편이 명확하기 때문이다.
- 근거: 사용자가 별도 저장소를 만들고 로컬에서 Codex로 사용하는 방식에 동의했다.
- 영향:
  - 로컬에 저장소를 clone한 뒤 해당 디렉터리에서 Codex를 실행한다.
  - 문서 변경은 Git으로 버전 관리한다.
  - 중요 의사결정은 이 파일에 기록한다.
- 재검토 조건:
  - 개발 저장소와 문서를 통합할 명확한 필요가 생길 때

# 가설

## 첫 사용자 직접 온보딩이 기능 추가보다 우선일 수 있다

- 상태: 가설
- 내용: 실제 사용자 한 명을 직접 가입시키고 관찰하는 것이 새로운 기능을 추가하는 것보다 빠르게 핵심 문제를 발견할 수 있다.
- 근거:
  - 현재 실제 가입 이탈 지점이 미확인이다.
  - 첫 사용자 확보가 최상위 목표다.
- 검증 방법:
  - 원장 또는 견주 후보 한 명의 가입 전 과정을 관찰한다.
  - 막힌 지점, 질문, 오류, 소요 시간을 기록한다.
- 확정 조건:
  - 직접 온보딩 계획을 사용자가 승인하거나 실제 수행했을 때

# 기록 규칙

1. 사용자가 확정하지 않은 제안을 `확정`으로 기록하지 않는다.
2. 실제 데이터와 해석을 구분한다.
3. 과거 결정이 바뀌면 삭제하지 않고 `대체됨`으로 남긴다.
4. 날짜와 이유를 반드시 남긴다.
5. 최신 결정과 기존 문서가 충돌하면 문서를 함께 수정한다.
6. 개인정보나 사용자 식별 정보는 필요한 범위 이상 기록하지 않는다.
